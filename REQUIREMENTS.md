# 要件定義: フローティング・ポモドーロタイマー

> [!success] 結論サマリ
> - **作る価値はある**: 「①ネイティブ軽量 ②全Space＋フルスクリーン上への完全追従 ③作業開始で自動透明化 ④無料/買い切り」の4点同時達成アプリは存在しない（monora が最も近いが②は未公称）
> - **技術選定は SwiftUI + AppKit（NSPanel）一択**: コア挙動は本機（macOS 26.4.1）で実機検証済み。Tauri はプラグイン依存＋private API、Electron はフルスクリーン上表示が信頼できず（refuted）、Web（Document PiP）は核心体験を再現できない
> - **最大のリスクは monora の全機能をコピーする scope creep**: MVP は「タイマー＋追従パネル」に絞り、環境音・統計チャート・カレンダー連携は意図的に捨てる

## 成果物

### 1. プロダクトコンセプト

**「作業画面を一切邪魔しない。でも、ちゃんとそこにいる」タイマー**（monora 開発者・梅本周作氏の言葉）を、Raycast Notes 級の追従性（全Space・フルスクリーン上）で実現する。

- タイマーは「強制する看守」ではなく「提案する秘書」。中断・延長・スキップに罪悪感を持たせない
- 存在感は3段階で制御する: **ホバー時=フル操作UI / 待機時=数字だけ / 作業中=透明化して溶け込む**
- ローカル完結・テレメトリなし・サブスクなし

### 2. リサーチで確定した事実（検証済み技術主張）

ワークフローの検証フェーズで、一次情報（Apple公式ドキュメント・SDKヘッダ・実機テスト・公式リポジトリ）に基づき確認した。**設計はすべてこの表の上に立っている。**

| # | 主張 | 判定 | 根拠の要点 |
|---|------|------|-----------|
| 1 | NSPanel(.nonactivatingPanel) + level .floating + collectionBehavior [.canJoinAllSpaces, .fullScreenAuxiliary] で**他アプリのフルスクリーンSpace上にも表示**できる | ✅ confirmed | **本機（macOS 26.4.1）で2プロセス実機テスト済み**。occlusionState.visible 維持・CGWindowList で前面確認・TCC プロンプトなし（特別権限不要）。テストコードは後述 |
| 2 | collectionBehavior フラグ**だけ**では不十分 | ⚠️ refuted（元主張が） | NSPanel であることが本質。通常の NSWindow に同じフラグを足してもフルスクリーン上に出ない実例あり（Firefox PiP / Apple Forums thread 26677）。NSPanel + .nonactivatingPanel + level + フラグの**3点セット**が必要 |
| 3 | クリックしてもフォーカスを奪わない（.nonactivatingPanel）は MAS サンドボックス内でも追加 entitlement 不要 | ✅ confirmed | 前例 = Maccy（MAS 販売中・OSS で実装確認可能）。※TomatoBar は nonactivatingPanel 不使用なので前例として不適 |
| 4 | LSUIElement（メニューバー常駐・Dock 非表示）でもアクションボタン付き通知を送れる | ✅ confirmed | UserNotifications は macOS 10.14+。前例 = TomatoBar。**注意**: デフォルトの「バナー」通知は数秒で消えボタンは畳まれる → 通知だけに 3択UX を依存させない |
| 5 | タイマーは tick 積算ではなく**終了予定時刻（Date）との差分**で計算しないとスリープ・App Nap でずれる | ✅ confirmed | Apple 公式「A timer is not a real-time mechanism」。Date 差分 + NSWorkspace.didWakeNotification 再描画で実用十分 |
| 6 | Raycast 拡張 API では独自フローティングウィンドウを作れない | ✅ confirmed | mode は view / no-view / menu-bar の3種のみ。要望 issue #3383 は2022年から放置。**Raycast 拡張案は不成立** |
| 7 | Tauri v2 はフルスクリーン上表示が標準 API で不可（コミュニティプラグイン tauri-nspanel 必須）、透過は private API フラグ必須（MAS 不可） | ✅ confirmed | tauri 本体に FullScreenAuxiliary のコードなし。公式スキーマに「private API は App Store に受理されない」と明記 |
| 8 | Electron の setVisibleOnAllWorkspaces(visibleOnFullScreen) はフォーカス操作なしでは信頼できない | ❌ refuted | バグ #36364 は**修正されずに stale クローズ**。公式に文書化された経路は type:'panel' だが Dock 強制非表示の副作用 issue も open のまま |
| 9 | Web（Chrome Document PiP）は他アプリのフルスクリーンSpace に出ない・透過/クリック透過不可・origin バー除去不可 | ✅ confirmed | Chromium ソースで FullScreenAuxiliary を設定する API を PiP が呼んでいないことを確認。透過要望 issue は3年放置。**「Webで v0」案は核心体験を検証できないため却下** |
| 10 | Raycast Notes は window level 8（.modalPanel）で浮いている | ✅ 実測 | 本機の Raycast 1.104.19 で CGWindowList 実測。**自作を .floating(3) にすると Raycast Notes と重なったとき必ず下になる**（レベル帯の序列は絶対） |

> [!warning] 未検証のまま残った主張（セッション上限によるエージェント失敗分）
> - monora 自身が「全Space追従・フルスクリーン上表示」を実装しているか → **公式情報に記載なし**（公称は「常に最前面」トグルと「スクリーン間移動の追従」まで）。¥300 の実機購入で5分で検証可能（→ §12）
> - monora の透明化がクリック透過（ignoresMouseEvents）を含むか → 未確認。本要件定義では MVP でクリック透過を採用しないため影響なし（→ §7）

### 3. ターゲットと利用シーン

- **一次ターゲットは自分自身**（配布は当面スコープ外 → §9）。エンジニアの深い作業（コーディング・執筆）中に、Space を頻繁に切り替え、エディタやブラウザをフルスクリーンで使う
- 解決する課題は **time blindness**: 「タイマーが動いていても、視界から消えると忘れる」（メニューバー型タイマー共通の弱点）。HN の ADHD 向け常時可視タイマーのスレッドでも「ポモドーロアプリの最大の問題は消えること」が核心課題として支持を得ている
- 同時に「目立ちすぎるタイマーは集中を奪い害になる」という研究知見もあり、**常時可視と非侵襲の両立**（= monora の透明化の正しさ）が設計の中心テーマ

### 4. 競合と差別化 — 自作する価値

リサーチの結論: **「①ネイティブ軽量 ②全Space＋フルスクリーン完全追従 ③作業開始で自動透明化 ④無料/買い切り・広告/同期なし」の4点を同時達成する既存アプリは確認できなかった。**

| アプリ | 形態 | 価格 | 弱点（自作との差分） |
|--------|------|------|---------------------|
| **monora** | 透明フローティング + メニューバー | ¥300 買い切り | 最有力参考。ただし全Space/フルスクリーン追従は**未公称**（公称は「常に最前面」まで） |
| **Focusmo** | フローティング（透明度/サイズ/位置調整可） | 無料（カスタム間隔は Pro $39/年） | 自動透明化なし・フルスクリーン上表示の保証なし |
| Session | Mini Player（最前面小窓） | $39.99/年 | サブスク疲れの典型。「タイマーごときに年40ドル」 |
| Flow | フローティング + メニューバー | freemium | 「透明すぎて存在を忘れる」苦情 = **視認性チューニング不全**。この生の声が3段階存在感制御の根拠 |
| TomatoBar (OSS) | メニューバーのみ | 無料 | フローティング窓なし。**機能セットの MVP ベンチマーク**として優秀（3.3k stars） |
| Pomatez (OSS) | Electron フローティング | 無料 | 非ネイティブで重い。「機能はあるが体験が違う」 |
| MaddaFloatingClock (OSS) | 最前面×全Space追従の時計 | 無料 | 製品の磨き込みなし。**技術的実現可能性の参照実装**として有用 |
| Raycast Focus | Raycast 本体機能 | Raycast 依存 | フローティング窓なし。拡張APIでは作れない（確定 → §2-6） |

**差別化の核は機能の数ではなく「追従挙動と存在感の制御」の2点。** 機能リストはむしろ TomatoBar 相当まで絞るのが正しい。

> [!info] 差別化の保険（monora 実機検証の結果次第の分岐）
> ¥300 の実機検証で monora が実は全Space＋フルスクリーン追従済みと判明した場合、差別化軸を「無料/OSS・グローバルホットキー召喚・セッションログを JSONL で吐いて自分の得意領域（Python/BigQuery）で分析」へスライドする。

### 5. 機能要件（MVP / v1 / やらないこと）

#### MVP（これだけで日常使用を開始する）

| # | 機能 | 仕様・根拠 |
|---|------|-----------|
| M1 | **フローティングパネル** | §7 の挙動仕様すべて。これが核 |
| M2 | **タイマーコア（2モード）** | 終了予定時刻（Date）ベース（§2-5）。**フローモード（主役・2026-06-11 追加要件）**: 作業はカウントアップで好きなだけ続け、止めた時点で「作業時間 × 比率」の休憩が自動算出される（Flowtime/Flowmodoro 方式。デフォルト比率 1:5 → 25分→5分、45分→9分。比率は 1:3〜1:6 で変更可、下限60秒）。作業中は「貯まった休憩時間」をライブ表示して動機づけにする。**クラシックモード（従）**: 従来の固定カウントダウン、作業/短休憩/長休憩/セット数の4値、デフォルト 25/5/15/4 |
| M3 | **自動開始の非対称トグル** | 休憩=自動開始（デフォルト ON、カウント漏れ防止）/ 次の作業=手動開始（デフォルト OFF、作業は意図的に始める）。独立した2トグル |
| M4 | **終了時の3択を1クリック** | 「休憩へ / +5分延長 / スキップ」。**延長・スキップを失敗扱いにしない**（abstinence violation effect 対策）。パネル UI を主、通知アクションを副とする冗長設計（§2-4 の通知バナー制約のため） |
| M5 | **終了の合図（2段構え）** | 第1=パネル自体の視覚変化＋サウンド（権限不要・macOS集中モード非依存）、第2=システム通知（アクションボタン付き）。「集中するためのアプリなのに集中モード中は終了に気づけない」矛盾への対処 |
| M6 | **メニューバー常駐** | MenuBarExtra + LSUIElement（Dock 非表示）。残り時間テキスト＋今日の完了セッション数を表示。パネルと視界を二重化する monora の設計を踏襲 |
| M7 | **グローバルショートカット** | 開始/一時停止、パネル表示/非表示の2つから |
| M8 | **データ保存（ローカルのみ）** | 設定= UserDefaults。セッション履歴= 1行1セッションの JSONL 追記（~/Library/Application Support/ 配下、schema: start/end/種別/完了フラグ）。チャートは作らず、分析は得意領域の Python/BigQuery に寄せる |

#### v0.7（2026-06-11 追加要件・実装済み）

| # | 機能 | 仕様・根拠 |
|---|------|-----------|
| M9 | **全画面休憩モード** | 休憩開始で全ディスプレイを覆うオーバーレイ（呼吸グロー＋巨大カウントダウン）。クリックは遮るがキーボード・メニューバー・Dock は奪わない。「+5分 / 小さく / スキップ」の逃げ道を常設（強制すると OFF にされて終わるため）。メニューでトグル可（デフォルト ON） |
| M10 | **作業メモ** | 進行中セッションに「何をしているか」を付けて JSONL に保存。入力はメニューバーのダイアログと API のみ（パネルにテキスト入力を置かない原則を維持） |
| M11 | **localhost API（Claude Code 連携）** | `127.0.0.1:51766` の HTTP API。status/stats/sessions 参照、start/pause/finish/memo/skip/extend/reset 操作。ループバック限定・認証なし（個人用途のトレードオフ。配布時はトークン認証必須）。`scripts/pomo` CLI ラッパ同梱。リポジトリの CLAUDE.md に API 仕様を記載し、Claude Code セッションが自動で連携可能 |
| M12 | **統計表示** | メニューバーに今日/今週の完了数・作業時間。パネルは待機中ホバーで今日の積み上げを表示。チャートは作らない方針を維持（生ログは API /sessions で取得可） |

> [!note] SaaS 化はしない（2026-06-11 の判断）
> 「覇権を取れる SaaS に」という要望に対し、クラウド・アカウント・課金化は §4 の差別化（ローカル完結・買い切り文化）を自壊させるため不採用。代わりに「SaaS 級の機能をローカルで」= 全画面休憩・統計・メモ・API 連携をすべてローカル実装する路線を取る。

#### v1（MVP が日常で回ってから）

- 透明化の磨き込み: ウィンドウリサイズ時の UI 比例スケール（monora が v1.1.1 で後付けした難所）
- ログイン時起動（SMAppService、トグル、デフォルト OFF）

#### やらないこと（理由付き・スコープクリープ防止柵）

| 機能 | 入れない理由 |
|------|-------------|
| 環境音 | 効果に強い個人差（メタ分析: 注意に問題のない人にはむしろ悪化）。音声再生は別系統の実装コスト。Spotify 等の併用前提 |
| ストリーク/連続記録 | 途切れた時に動機を大きく毀損する（ストリーク研究）。罪悪感で離脱を生む |

| Google カレンダー連携 | OAuth 実装が単体で MVP 超えの工数。完全オフラインの価値を壊す |
| クラウド同期/アカウント | ローカル完結の価値と矛盾 |
| サイト/アプリブロック | 権限要求が重くミニマルさと矛盾 |
| Mac App Store 配布 | 当面は自分専用（§9）。配布したくなったら別案件として起こす |

### 7. フローティング挙動の仕様（核心）

#### A. ウィンドウ実装レシピ（実機検証済み）

```swift
// 検証済みレシピ（macOS 26.4.1 実機、2プロセステストで他アプリのフルスクリーン上表示を確認）
let panel = NSPanel(contentRect: ..., styleMask: [.nonactivatingPanel, .borderless], ...)
panel.level = .floating                                          // 常時最前面（§12-1 のレベル論点あり）
panel.collectionBehavior = [.canJoinAllSpaces, .fullScreenAuxiliary]  // 全Space追従＋フルスクリーン上
panel.becomesKeyOnlyIfNeeded = true                              // キーフォーカスを奪わない
panel.hidesOnDeactivate = false
panel.isOpaque = false
panel.isMovableByWindowBackground = true                         // 全面ドラッグ
NSApp.setActivationPolicy(.accessory)                            // LSUIElement 相当
// SwiftUI ビューは NSHostingView で載せる
```

- **NSPanel であることが本質**（通常の NSWindow + フラグでは不可。§2-2）
- **パネル内にテキスト入力 UI を置かない**（needsPanelToBecomeKey=true のビューがあるとキーフォーカスがパネルに移る。タイマーはボタン操作のみで完結させる）
- Mission Control 中は floating レベルのウィンドウは自動的に隠れる（仕様。「邪魔をしない」方向なので許容）
- **既知の罠（実機テストで観測済み）**: フルスクリーンSpace への遷移/復帰時にシステムがパネル位置を勝手に移動させる。位置の保存・復元ロジックが必須

#### B. 存在感の3段階制御（プロダクトの魂）

| 状態 | トリガー | 表示 |
|------|---------|------|
| 操作 | ホバー中 | フル UI（タブ＋数字＋3ボタン）、不透明度 1.0 |
| 待機 | マウスが離れた | 数字＋進捗バーのみ |
| 集中 | タイマー実行中（ホバーなし） | 全体の不透明度を下げて溶け込む（デフォルト 0.3、ユーザー調整可）。ホバーで即 1.0 に復帰 |

- 遷移はすべて 300-600ms の ease-out フェード
- **クリック透過（ignoresMouseEvents）は MVP では採用しない**。true にするとホバーイベントも届かなくなり「ホバーで復帰」のトリガー設計が壊れるため。グローバルマウス監視で両立させる案は v2 の研究項目
- Flow の苦情「透明すぎて存在を忘れる」への対策として、透明度の下限はユーザーが設定できるようにする

#### C. 表示コンテキスト対応表（完全性批評の指摘を反映）

| コンテキスト | 対応 | 根拠 |
|--------------|------|------|
| 通常 Space（複数） | Must | .canJoinAllSpaces（検証済み） |
| 他アプリのフルスクリーン Space | Must | .fullScreenAuxiliary + NSPanel（検証済み） |
| Mission Control | Won't（隠れてよい） | floating レベルの仕様挙動 |
| Stage Manager | Should（受け入れテストで確認） | 未検証。macOS 13+ の .canJoinAllApplications 併用を必要なら検討 |
| 複数ディスプレイ | Must（1ウィンドウのみ） | 位置を frameAutosave 相当で保存、復元時に visibleFrame 内へ補正。NSApplication.didChangeScreenParametersNotification で画面外に出たら主ディスプレイへ戻す |

### 8. 技術選定

#### 推奨: SwiftUI + AppKit（NSPanel）ハイブリッド — monora と同構成

| 評価軸 | Swift/AppKit | Tauri v2 | Electron | Wails/Fyne | Raycast拡張 | Web (Doc PiP) |
|--------|:---:|:---:|:---:|:---:|:---:|:---:|
| ①常時最前面 | ○ | ○ | ○ | △ | ✕ | ○ |

**推奨理由**: 要件①〜⑥すべてを first-party API だけで満たすのは Swift/AppKit と Electron のみで、Electron は③が refuted（§2-8）かつ常駐タイマーには重量過剰。本件の難所は「言語」ではなく「ウィンドウ挙動」にあり、*『言語は得意だがウィンドウ挙動はネイティブ層のバグと戦う』（Electron/Tauri）vs『言語は未経験だがウィンドウ挙動は OS 公式 API そのもの』（Swift）* のトレードオフで後者が有利。実装面積は「NSPanel サブクラス1つ＋タイマーロジック＋MenuBarExtra＋設定画面」と狭く、定石コード（Cindori チュートリアル・FreeCastNotes・Maccy・TomatoBar 等の OSS）が豊富なため AI 支援開発と相性が良い。monora 自身が「デザイナーが Claude Code で SwiftUI ネイティブを個人開発した」前例。

**次点**: Electron（API は最も明示的・TS で書ける）。ただし採用するなら type:'panel' での30分プロトタイプ検証を通過した場合のみ。

**この選定が破綻しうる条件**:
1. **macOS メジャーアップデートでウィンドウ挙動が変わる** — iTerm2 のコードコメントに macOS 26 でのレベル挙動変更が実記録されている。受け入れテスト（§10）を OS アップデートごとに再実行する運用で緩和
2. **Swift 学習コストをゼロと見積もる** — ビルドエラー（署名・Info.plist・@MainActor）で自力判断できず詰むのが挫折の最頻シナリオ。緩和策: (a) Xcode GUI に依存しない CLI ビルドループ（swift build / xcodebuild）を最初に確立し Claude Code が自走できる形にする (b) pbxproj 直編集を避け SPM executable target か XcodeGen で宣言的に管理 (c) スパイクコードが既にある（§11 Step 0）
3. **パネルにテキスト入力を追加したくなる** — needsPanelToBecomeKey の境界条件でフォーカス挙動が崩れる。要件として「パネルはボタン操作のみ」を維持する

### 9. 非機能要件

| 項目 | 要件 |
|------|------|
| リソース | アイドル時 CPU 1% 未満・メモリ 100MB 以下（ネイティブなら 20-50MB が常識値）。タイマー表示の更新は秒1回に制限 |
| 時刻管理 | 終了予定時刻（Date）ベース＋ NSWorkspace.didWakeNotification で復帰時に再計算。**5分以上のスリープを跨いだ作業セッションは「中断」扱い**（継続でも失効でもなく、罪悪感なしの中立な記録）。時計変更が気になる場合の最堅牢解は mach_continuous_time（スリープ中も進む単調クロック）だが MVP は Date 差分で十分 |
| 通知権限 | 初回タイマー完了の直前に要求（起動直後に求めない）。拒否されてもパネルの視覚変化＋サウンドで全機能が成立する設計（M5） |
| データ | 完全ローカル。外部送信なし。クラッシュ時に進行中セッションは破棄してよい（記録は完了時のみ追記） |
| 配布 | **当面は自分の Mac のみ**。ad-hoc 署名のローカルビルドで完結（Apple Developer Program 不要）。他人に配布する時点で Developer ID 署名＋ notarization（$99/年）を別案件として起こす。※macOS Sequoia 以降、未署名アプリの Ctrl+クリック起動バイパスは廃止済み（自己ビルドなら無関係） |
| 対象 OS | 開発機の macOS（26.x）のみサポート。後方互換は追わない（MenuBarExtra は macOS 13+、参考に monora は 14+） |

### 10. 受け入れ基準（手動チェックリスト）

Claude Code は GUI 挙動を直接確認できないため、フローティング挙動の検証は人間の手動テストが必須。各マイルストーンの完了条件として以下を実行する。

- [ ] 1. 別アプリ（VSCode / Safari）を**ネイティブフルスクリーン**にしてもタイマーが上に表示される
- [ ] 2. Space を切り替えてもタイマーがついてくる
- [ ] 3. タイマーのボタンを操作した直後、**エディタへのタイピングが途切れない**（フォーカス非奪取。メニューバーのアプリ名が変わらないことでも確認可）
- [ ] 4. Mac の蓋を閉じて30分後に開く → 残り時間が実時間と一致している（または中断ポリシー通りの表示）
- [ ] 5. アクティビティモニタでアイドル時 CPU 1% 未満・メモリ 100MB 以下
- [ ] 6. 通知 OFF＋集中モード ON でも、セッション終了に気づける（パネルの視覚変化＋サウンド）
- [ ] 7. 外部ディスプレイを抜いてもウィンドウが画面外に消えない
- [ ] 8. Mission Control / Stage Manager での挙動を確認し、想定（§7-C）と一致する

### 12. 未解決の論点

1. **ウィンドウレベルの選択**: .floating(3) だと Raycast Notes(8) と重なったとき下になる（§2-10）。MVP は .floating で開始し、Raycast Notes と併用して困ったら rawValue 9 か .statusBar(25) に上げる。上げすぎると通知や他 UI と競合するトレードオフがあるため実使用で判断
2. **monora の実機検証（¥300・5分）**: 設定で Always on Top を有効 → 別 Space へスワイプ → 任意アプリをフルスクリーン化、で追従を確認。結果次第で差別化軸を §4 の保険に切り替え。デザインの実物観察（透明化の速度感・ホバー挙動）としても価値があるので**購入推奨**
3. **クリック透過の将来対応**: ignoresMouseEvents とホバー復帰の両立はグローバルマウス位置監視が必要。v2 の研究項目
4. **アプリ名**: 未定。monora が Mono(ひとつの) + Ora(時間) の造語である命名アプローチは参考になる
5. **フォントライセンス**: Inter は SIL OFL でほぼ確実に同梱可、LINE Seed JP は未確認。MVP はシステムフォントなので急がない
6. **Doc PiP のタイマースロットリング**: 検証エージェントがセッション上限で失敗し未検証のまま。ただし Web 案自体を却下したため現状影響なし
